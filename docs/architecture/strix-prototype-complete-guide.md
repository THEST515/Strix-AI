# Strix 原型完整讲解

> 本文用于解释 Strix 开源项目的工作机制，以及本平台对 Strix 的实际集成方式。
>
> 重要边界：本文根据本项目的实际集成方式解释 Strix 的原型工作机制。Strix 的具体内部实现、工具集合和提示词细节应以其开源仓库与当前版本代码为准。文中会明确区分“Strix 原生能力”和“本项目平台层能力”，避免把平台封装误说成 Strix 内部机制。

## 1. 一句话理解 Strix

Strix 是一个以大语言模型为决策核心、以浏览器和 HTTP 等工具为执行手段、通过多轮观察与反馈进行黑盒安全分析的 AI 安全测试代理。

它不是简单的“把网页丢给模型总结”，也不是传统意义上只依赖固定规则的扫描器。它更接近一个能够自主规划动作的安全测试工程师：

```text
观察目标
  -> 理解页面和攻击面
  -> 规划下一步测试
  -> 调用浏览器或请求工具
  -> 分析响应和运行反馈
  -> 继续探索、收缩或验证
  -> 整理漏洞证据
```

## 2. Strix 解决什么问题

传统自动化扫描器通常依赖：

```text
固定规则 + 预定义请求 + 特征匹配
```

这种方式在稳定、结构化的输入上效率较高，但面对现代 Web 应用时会遇到困难：

- 页面结构经常变化
- 一个漏洞可能需要多个页面和多个请求配合
- 登录状态和权限关系会影响结果
- 参数的安全意义依赖业务上下文
- 很多接口没有明显的静态特征
- 漏洞验证需要根据响应动态调整测试方法

Strix 的目标是利用 LLM 对页面、请求、响应和日志进行上下文理解，再决定下一步动作，从而覆盖更复杂的黑盒测试流程。

## 3. Strix 的核心组成

可以把 Strix 原型抽象成六个部分：

| 组成部分 | 作用 |
|---|---|
| 目标输入层 | 接收网站、Web 应用、API 或本地代码等扫描目标 |
| 运行控制层 | 解析 CLI 参数、创建运行、管理模式与目标 |
| Agent 决策层 | 让 LLM 根据当前上下文制定下一步安全测试动作 |
| 工具执行层 | 执行浏览器操作、HTTP 请求、重放、读取和分析等动作 |
| 证据与状态层 | 保存日志、运行状态、LLM 使用量、漏洞和摘要 |
| 报告生成层 | 把已验证结果整理为漏洞报告、证据和修复建议 |

这些模块的具体名称可能随 Strix 版本变化，但从原型机制上可以这样理解。

## 4. Strix 的 Agent Loop

Strix 最核心的机制是“感知、决策、行动、反馈”循环。

### 4.1 感知

Strix 首先收集目标环境中的信息，例如：

- 页面标题和正文
- 链接和路由
- 表单、输入框和按钮
- URL 查询参数
- API 路径和请求方法
- Cookies、认证状态和会话信息
- 文件上传入口
- 页面脚本或接口响应中的线索
- HTTP 状态码、响应头和响应正文

这些信息构成当前 Agent 的上下文。

### 4.2 决策

LLM 根据当前上下文判断：

```text
看到了什么？
哪些位置可能存在安全风险？
哪个入口值得优先测试？
下一步应该用什么工具？
测试输入应该如何构造？
当前结果是否足以形成证据？
```

例如，Strix 发现一个 `id=1` 参数时，可能会考虑：

- 修改为其他对象 ID，观察是否越权
- 使用不存在的 ID，观察错误处理
- 切换认证状态后比较响应
- 检查是否存在水平或垂直权限差异

这里的重点不是某个固定 payload，而是 LLM 根据参数语义和响应上下文选择动作。

### 4.3 行动

LLM 本身不能直接点击浏览器或发送网络请求，因此 Strix 提供工具接口。LLM 生成工具调用意图，Strix 负责执行。

典型工具动作包括：

- 打开页面
- 点击链接或按钮
- 填写和提交表单
- 发送 HTTP 请求
- 重放已有请求
- 修改请求参数
- 比较不同认证状态下的响应
- 读取页面或响应内容
- 保存测试证据

### 4.4 反馈

工具执行后会返回新的反馈：

- 请求是否成功
- 页面是否跳转
- 响应状态码是否变化
- 输入是否被反射
- 是否出现异常或错误信息
- 是否访问到了不属于当前用户的数据
- 是否形成了可复现的安全影响

新的反馈会进入下一轮上下文，Agent 再次做决策。

### 4.5 循环终止

一次扫描通常在以下情况下结束：

- 形成了漏洞报告
- 目标攻击面已经探索完毕
- 没有发现可验证漏洞
- 扫描超时
- 运行环境失败
- 用户取消扫描
- Agent 无法继续执行有效动作

值得注意的是，“循环结束”和“全面完成”不是同一个概念。扫描结束只说明当前运行停止，不代表目标不存在遗漏漏洞。

## 5. 从攻击面到漏洞证据

Strix 的漏洞发现不是单一步骤，而是一个逐渐收敛的过程。

### 5.1 攻击面识别

首先要知道目标有哪些可测试入口：

```text
页面
表单
输入参数
API 接口
认证点
文件上传点
对象标识符
外部请求入口
```

攻击面越完整，后续测试覆盖率越高。

### 5.2 可疑信号发现

在攻击面上进行探索和测试，观察是否出现：

- 输入被危险地反射
- 未授权用户获得了受限数据
- 参数修改改变了对象访问范围
- 服务端向外部地址发起请求
- SQL、模板或命令解释行为异常
- 文件路径处理突破了预期目录
- 上传内容被当作可执行资源处理

此时通常只能称为“可疑信号”或“候选问题”。

### 5.3 定向验证

Strix 需要继续验证：

- 是否可以重复触发
- 是否确实由攻击者可控输入造成
- 是否存在真实安全影响
- 是否受认证、权限或业务条件限制
- 是否只是正常业务行为或误报

### 5.4 证据整理

高质量漏洞报告至少应尽量包含：

- 漏洞标题
- 风险等级
- 目标入口
- 请求方法和关键参数
- 复现步骤
- 实际响应或行为
- 影响说明
- 修复建议

因此，Strix 的目标不只是“猜测一个漏洞名称”，而是尽量形成可复现、可解释、可交付的漏洞证据。

## 6. 常见漏洞类型与测试思路

下面是原型层面的测试方向，不代表 Strix 每次都会覆盖全部类型。

| 漏洞方向 | 典型关注点 |
|---|---|
| XSS | 输入是否进入 HTML、脚本、属性或 DOM 危险上下文 |
| SQL 注入 | 参数是否影响查询语义、错误信息或结果集 |
| IDOR / 越权 | 修改对象 ID 后是否能读取或修改他人资源 |
| 认证问题 | 未登录、低权限和高权限状态下的访问差异 |
| SSRF | 服务端是否根据用户输入访问外部或内部地址 |
| 文件上传 | 文件类型、路径、存储位置和执行行为 |
| 路径穿越 | `../` 等输入是否突破预期资源目录 |
| SSTI | 用户输入是否进入模板解释上下文 |
| CSRF | 敏感操作是否缺少请求来源或令牌校验 |
| Open Redirect | 重定向目标是否完全受用户输入控制 |
| 敏感信息暴露 | 响应、错误和静态资源是否泄露敏感数据 |

实际漏洞覆盖取决于目标、认证状态、工具可用性、扫描时间、LLM 判断和测试策略。

## 7. Strix 如何尽量提高全面性

Strix 不能保证漏洞发现绝对全面，但可以通过以下机制扩大覆盖面：

### 7.1 攻击面枚举

尽量识别页面、链接、表单、参数、API、认证点和上传点。

### 7.2 上下文驱动测试

根据页面和业务语义选择测试方向，而不是对所有参数机械使用同一组规则。

### 7.3 浏览器与 HTTP 结合

浏览器适合处理真实交互和会话，HTTP 工具适合精确修改请求、重放和比较响应。两者结合能够覆盖更多多步骤流程。

### 7.4 多轮反馈

一次测试的结果会影响下一步动作，Agent 可以沿着有价值的线索深入。

### 7.5 候选问题复现

对可疑信号继续验证，减少仅凭异常响应产生误报。

### 7.6 运行时间与策略约束

如果扫描一直停留在开放探索阶段，需要限制范围、调整 instruction 或把测试收缩到已识别的输入点和接口。

## 8. 为什么不能宣称“全面扫描”

以下因素都会造成遗漏：

- 隐藏页面或接口没有被发现
- 登录流程无法完成
- 缺少足够权限或业务数据
- 动态状态无法构造
- 扫描时长不足
- LLM 选择了某些路径而忽略其他路径
- Agent 在侦察阶段反复探索
- 浏览器、Docker 或模型 API 发生异常
- 某些漏洞需要人工理解业务规则
- 当前策略没有覆盖对应漏洞类型

因此应使用更准确的表述：

> Strix 通过自主攻击面探索、上下文推理、多工具操作和多轮验证，尽量提高黑盒安全分析的覆盖率与结果可信度，但不能证明目标不存在未发现漏洞。

## 9. Strix 的运行产物

在本项目中，Strix 的真实运行产物位于：

```text
strix_runs/<run>/
```

常见文件和信息包括：

| 产物 | 用途 |
|---|---|
| `run.json` | 运行 ID、目标、状态、时间、LLM 使用量、摘要等 |
| `strix.log` | 原始运行日志和工具调用轨迹 |
| `vulnerabilities.json` | 当前运行写出的漏洞发现 |
| `scan_results` | 扫描摘要、技术分析和修复建议 |
| `finding_translations.zh-CN.json` | 本平台生成的中文翻译缓存，不是 Strix 原生产物 |

`finding_translations.zh-CN.json` 是平台层扩展，用于避免同一个真实 run 在刷新后反复触发翻译或出现中英文切换。

## 10. Strix CLI 与本项目的实际调用链

本项目的真实链路如下：

```text
前端选择“真实 Strix 扫描”
  ↓
GET /api/preflight?target=...
  ↓
检查目标、Docker、Strix CLI、LLM 配置
  ↓
创建 latest_real_run 任务
  ↓
点击“启动真实 Strix 扫描”
  ↓
后端再次执行 preflight
  ↓
构造 strix -n --target ... --scan-mode ...
  ↓
以后台子进程启动 Strix
  ↓
Strix 写入 strix_runs/
  ↓
平台轮询 /runtime 和 /results
  ↓
读取日志、运行状态、攻击面、LLM 用量和 findings
  ↓
中文化并展示或导出报告
```

后端实现重点：

- `src/backend/services/strix_runner.py`：CLI 启动、等待、取消和失败判定
- `src/backend/services/real_run_loader.py`：运行产物读取和结果归一化
- `src/backend/services/runtime_analyzer.py`：结构化运行态解释
- `src/backend/services/preflight.py`：运行前环境检查
- `src/backend/api/demo_server.py`：API 和任务生命周期编排

## 11. Strix 原生能力与平台增强的边界

### 11.1 更接近 Strix 原生能力的部分

- LLM 驱动的安全测试决策
- 浏览器和 HTTP 等工具操作
- 黑盒探索和交互
- 运行日志
- 运行目录和漏洞产物
- 扫描摘要与漏洞报告形成

### 11.2 本项目平台层能力

- 任务创建和任务列表
- `fixture` 与 `latest_real_run` 双路径
- 运行前检查
- 后台运行、超时和取消管理
- stale-success 防护
- 失败或超时后的部分 findings 保留
- 五阶段运行态解释
- 攻击面与收敛诊断
- 中文规则、LLM 翻译和翻译缓存
- Markdown / DOCX 导出
- 中文产品化界面

这一区分对技术理解非常重要。不能把平台层的 `preflight`、`convergence` 或 DOCX 导出称为 Strix 自身能力；它们是围绕 Strix 建立的工程封装。

## 12. 为什么本项目不直接修改 Strix 内部

本项目选择通过 CLI 和运行产物适配 Strix，原因是：

1. 保留真实 Strix 链路，避免重写扫描器。
2. 降低与 Strix 内部实现的耦合。
3. Strix 升级时主要调整适配层。
4. 扫描引擎与平台层的职责边界更清楚。
5. `fixture` 可以独立验证前端和报告链路。

架构边界是：

```text
Strix 负责发现和验证
平台负责编排、解释、中文化和交付
```

## 13. 本项目的收敛式运行态

开放式 Agent 扫描存在一个常见问题：它可能持续探索，但没有形成最终漏洞证据。

因此平台把一次运行解释为以下阶段：

```text
preflight
  ↓
recon
  ↓
surface_analysis
  ↓
targeted_validation
  ↓
evidence_packaging
  ↓
completed / failed / timeout / cancelled
```

平台会从 `run.json`、`strix.log`、当前任务状态和已落盘 findings 中估算：

- 当前阶段
- 页面、表单、参数、API、认证和上传攻击面
- LLM 请求量和 Token 使用量
- 是否已有疑似问题或已验证漏洞
- 最近是否还有有意义的运行事件
- 当前是否收敛
- 下一步建议继续、收缩还是结束

这不是对 Strix 内部 Agent 状态的直接读取，而是平台基于外部运行证据进行的解释层。

## 14. 研究 Strix 时建议观察什么

如果要深入研究 Strix 原型，建议重点观察以下内容：

### 14.1 输入

- CLI 参数
- 目标类型
- 扫描模式
- instruction
- LLM 模型和配置

### 14.2 决策

- LLM 每轮收到哪些上下文
- 如何选择工具
- 是否存在最大轮数或时间约束
- 什么时候决定继续探索
- 什么时候决定生成漏洞报告

### 14.3 工具

- 浏览器工具如何维护会话
- HTTP 请求如何重放
- 工具失败如何反馈给 Agent
- 工具权限和 Docker 边界是什么

### 14.4 证据

- 哪些事件会写入日志
- 什么时候写入 vulnerability report
- `run.json` 的状态如何变化
- 运行中是否会先写出部分 findings
- 失败时产物是否仍然可用

### 14.5 可靠性

- 同一目标多次扫描结果是否一致
- 扫描成本和 Token 数量
- 长时间无进展时的行为
- Docker、浏览器和 LLM 任一组件失败时的表现
- 结果是否能够被独立复现

## 15. 核心概念速览

Strix 与本平台的关系可以概括为以下三点：

> Strix 是一个以大语言模型为决策核心的黑盒安全测试代理。
>
> 它通过观察目标、规划测试动作、调用浏览器和 HTTP 工具、分析反馈并循环验证，尝试形成可复现漏洞证据。
>
> 本平台不替代 Strix，而是在其上增加任务编排、运行态解释、中文风险整理和报告导出，使真实扫描链路更容易观察和交付。

## 16. 最终结论

Strix 的原型本质是：

```text
LLM 决策
+ 工具调用
+ 黑盒反馈
+ 多轮探索
+ 漏洞证据验证
= AI 辅助安全测试代理
```

它的核心价值是处理复杂、动态、依赖上下文的 Web 安全测试问题；它的核心限制是自主探索具有不确定性，不能天然保证全面性、稳定性和零误报。

本项目的工程价值在于把这种能力封装为一个边界清晰的产品闭环：

```text
任务配置
  -> 运行前检查
  -> 真实 Strix 执行
  -> 运行态解释
  -> 风险中文化
  -> 报告导出
```
