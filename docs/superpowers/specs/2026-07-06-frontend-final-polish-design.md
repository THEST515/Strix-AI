# 前端终版打磨设计

## 背景

当前项目是“基于 Strix 的 AI 辅助安全分析平台”课程演示型 MVP，后端最小闭环和前端功能链路已经可用，但前端整体观感仍偏“概念控制室 + dashboard 混合体”，不够像成熟产品。  
本轮目标不是补主功能，而是在不破坏现有 API 对接、显式 `run`、Markdown 导出、`fixture` / `latest_real_run` 路径的前提下，把页面收束到更适合答辩录屏的成品状态。

## 设计结论

最终方向定为：`A1 = 深色、冷静、产品级演示`。

一句话定义：

> 页面应首先让人感到“这是一个已经能演示和交付的产品界面”，其次才是安全分析题材的气质表达。

## 设计目标

1. 保留深色基底与安全分析场景的专业感。
2. 去掉过强的“控制室戏剧化氛围”和概念化台词。
3. 优先提升演示效率，让老师第一眼看懂用途、流程和结果。
4. 不改变现有功能结构，不让用户重新学习页面操作。
5. 保持当前前端架构和后端 API 对接能力不回退。

## 不在本轮范围内

1. 不新增新业务功能。
2. 不调整后端接口语义。
3. 不重做任务创建、显式运行、导出、结果加载的交互逻辑。
4. 不在本轮处理所有真实 `latest_real_run` 文本的全文翻译问题。

## 现状问题

### 1. 风格混杂

当前界面同时存在三种语言：

1. “控制室”氛围层。
2. “课程演示工具页”说明层。
3. “成熟产品界面”功能层。

这三层没有完全统一，导致观感上仍然像一次中间态方案，而不是最终交付界面。

### 2. 首屏表达不够产品化

当前首屏虽然功能正确，但仍保留较强的氛围性包装，导致第一眼的信息优先级不够稳定。

### 3. 面板厚重感过强

大量玻璃感、光效和强边框会让页面更像概念视觉稿，而不是能长期使用的实际产品。

### 4. 风险详情还不够像评审界面

风险详情虽然已经可用，但版式仍可进一步向“问题审阅 / 结果讲解”靠拢，而不是继续强调视觉装饰。

## 目标界面原则

### 1. 产品先于氛围

视觉优先级应当是：

1. 功能模块
2. 状态层级
3. 安全分析题材氛围

而不是先营造气氛，再承载功能。

### 2. 信息块真实可讲

每个区域都要支持口头讲解：

1. 左侧用于“怎么创建任务”
2. 中间用于“怎么看当前结果”
3. 右侧用于“怎么切换历史任务”
4. 下方用于“怎么讲风险详情”

### 3. 动效要有理由

只保留这三类动效：

1. 页面进入时的轻量 reveal
2. hover / focus / active 的交互反馈
3. 任务切换和状态变化时的微过渡

不保留只为“看起来炫”的氛围运动。

### 4. 深色但不压抑

视觉基底保持深色，但改成更稳定的产品深炭色系，而不是偏概念海报的高氛围深色。

## 具体改造方案

## 顶栏

### 目标

让顶栏更像真实产品 header，而不是沉浸式场景入口。

### 调整

1. 降低顶栏体积和装饰密度。
2. 只保留品牌、当前状态、结果来源等核心信息。
3. 去掉多余的舞台化视觉暗示。

## 首屏摘要区

### 目标

让用户一进入页面就知道：

1. 这是做什么的
2. 现在能演示什么
3. 应该从哪里开始

### 调整

1. 标题语言改为产品式表达，而不是概念叙事。
2. 保留一句功能摘要。
3. 保留两个清晰入口，如“新建任务”“查看结果”。
4. 将“功能 / 来源 / 用途”三条硬信息做成更简洁的产品摘要条。

## 主体三区

### 左侧：任务创建

目标是“工作台感”而不是“氛围表单”。

调整：

1. 表单容器更干净。
2. 标签、输入框、按钮统一成稳定的产品控件语言。
3. 说明文案缩短，只保留对演示有帮助的顺序说明。

### 中间：当前结果

目标是“结果总览台”。

调整：

1. 当前任务、目标地址、结果来源三项信息做成更轻量的结果摘要。
2. 显式 `run` 和导出按钮更像正式产品操作按钮。
3. summary strip 更强调数字和级别层级。
4. AI 摘要区改成更像结果说明面板，而不是 narrative panel。

### 右侧：任务列表

目标是“任务历史 / 结果切换”。

调整：

1. 降低单条任务卡片的装饰感。
2. 提高任务名称、来源、状态三者的清晰度。
3. 强化 active task 的产品态选中效果。

## 风险详情区

### 目标

更像成熟产品里的 issue review / finding review 区。

### 调整

1. 减少舞台化装饰线条。
2. 标题、严重级别、摘要、证据、修复建议重新整理层级。
3. 让每条 finding 更适合口头逐条讲解。
4. 严重级别颜色仅在级别上强调，不扩散成整卡强氛围。

## 配色与组件语言

## 基础配色

1. 主背景：深炭黑 / 冷灰黑。
2. 主强调色：单一冷蓝。
3. 风险色：仅用于 `严重 / 高危 / 中危 / 低危 / 提示`。
4. 中性色：统一冷灰，不混用过多暖色。

## 组件规则

1. 降低玻璃感和 blur 强度。
2. 边框更细、更稳定。
3. 圆角体系收敛到一套产品级半径。
4. 阴影只用于层级，不用于渲染戏剧感。

## 文案策略

1. 不写“假大空”的概念句。
2. 所有标题优先说明用途。
3. 所有辅助文案优先服务讲解顺序。
4. 中文表达保持短句、明确、可读。

## 文件影响范围

本轮预计主要修改：

1. `src/frontend/index.html`
2. `src/frontend/styles.css`
3. 必要时微调 `src/frontend/rendering.mjs`

默认不改或尽量少改：

1. `src/frontend/app.js`
2. `src/frontend/taskData.mjs`
3. 后端与测试接口语义

## 数据流与行为约束

1. 保持当前 `create task -> load results -> explicit run -> export` 行为不变。
2. 保持 `fixture` 与 `latest_real_run` 两条来源路径不变。
3. 保持当前浏览器 smoke test 能覆盖的交互路径不变。
4. 所有视觉重构必须兼容现有 DOM 查询与渲染函数，若结构变更则同步最小化修正测试。

## 错误处理与状态展示

1. 空状态继续保留，但语气更产品化。
2. 表单反馈、运行反馈、导出反馈保持存在。
3. disabled 状态、hover 状态、active 状态都要明确可辨。

## 验证方案

### 代码验证

1. `node --check src/frontend/app.js`
2. `node --test tests/frontend/rendering.test.mjs tests/frontend/taskData.test.mjs`

### 页面验证

1. 本地 demo server 正常打开页面
2. `fixture` 路径正常创建和展示
3. `latest_real_run` 路径不回退
4. 显式运行按钮正常
5. Markdown 导出正常

### 观感验收

通过以下问题判断是否达到目标：

1. 第一眼是否更像成熟产品，而不是概念控制室？
2. 首屏是否能立即讲清楚页面用途？
3. 各区域是否更适合答辩时按顺序讲解？
4. 风险详情是否更像结果审阅界面？

## 实施顺序

1. 先重构 `index.html` 的文案与结构细节。
2. 再重构 `styles.css` 的色彩、层次、组件与动效。
3. 如结构变化影响渲染挂载点，再最小化调整 `rendering.mjs`。
4. 运行前端测试与语法检查。
5. 本地页面复看并做最后小修。
