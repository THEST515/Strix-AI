# 故障口径

## 1. 页面能开，但真实扫描点了没结果

优先口径：

> 前端链路是通的，当前问题通常不在页面，而在 Docker、Strix CLI 或模型配置。

立即检查：

1. `docker info`
2. `strix --help`
3. `STRIX_LLM`
4. `LLM_API_KEY / DEEPSEEK_API_KEY`

## 2. 扫描超时但没有漏洞

建议口径：

> 这表示本轮真实扫描没有在给定时长内收敛到可保留证据，不代表系统没有运行。需要结合执行轨迹区看它停在侦察、攻击面分析还是定向验证阶段。

## 3. 任务失败但页面还有结果

建议口径：

> 平台会尽量保留扫描过程中已经落盘的 findings，所以“任务失败”不一定等于“没有结果”。当前页面展示的是已保留的有效部分结果。

## 4. 结果一会英文一会中文

当前口径：

> 这轮已经做了两层收口：先走规则表，再走 LLM 翻译，并把翻译结果按真实 run 做持久化缓存。正常情况下，同一个 run 不会再来回跳版本。

## 5. 为什么有时说已完成，有时说失败

建议口径：

- 无 findings 且真实运行异常：显示失败
- 已保留 findings：按已完成展示结果

一句话解释：

> 如果结果已经保留下来，产品层面更适合按“执行完成”讲解；如果本轮什么都没留下，才按纯失败处理。

## 6. Docker daemon 权限或环境异常

建议口径：

> 当前代码链路已验证通过，但真实扫描依赖本机 Docker daemon。若 Docker 本身不可用，平台会明确失败，而不会再伪装成旧结果成功。

## 7. 现场兜底顺序

如果真实扫描现场不稳定：

1. 先切回 `fixture`
2. 继续展示摘要、风险详情、导出能力
3. 再说明真实扫描所依赖的外部环境项

## 8. 不该说的话

避免：

- “这是前端 bug”
- “Strix 没有被调用”
- “没有漏洞就说明没跑”
- “失败了所以这次没有任何有效输出”

更准确的说法：

- “链路已打通，当前是外部运行环境问题”
- “本轮未收敛到漏洞证据”
- “结果已部分保留”
